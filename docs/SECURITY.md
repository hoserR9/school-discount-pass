# Del Norte Nighthawk Discount Card — Security Overview

## For Sponsors and Board Members

This document explains how the Nighthawk Discount Card system protects cardholders, sponsors, and data.

---

## Card Security

### Unique Cryptographic IDs
- Every card gets a **unique 11-character ID** (e.g., DN-8F3A2B1C) generated using cryptographic random bytes
- No two cards can ever have the same ID
- Card IDs cannot be guessed or predicted

### Authentication Tokens
- Every digital card includes a **64-character authentication token** (also cryptographically generated)
- Apple Wallet uses this token when communicating with our server
- Tokens are never shown to the cardholder or anyone else
- Only Apple and our server know the token

### Anti-Screenshot Protection
- Apple Wallet passes have a **live barcode that animates** — screenshots capture a static image
- When scanned, the verification screen shows the **cardholder's name in large text** — cashiers can verify identity
- Screenshots of the QR code still work for scanning, but the verification screen will show the registered holder's name, not the screenshot-taker's name

### Anti-Sharing Protection
- **Rate limiting**: Cards scanned 5+ times in 24 hours are automatically flagged with a yellow warning screen
- **Rapid scan detection**: If the same card is scanned at a different business within 5 minutes, the cashier sees a warning
- **Deactivation**: Admins can deactivate any card instantly from the dashboard. The card immediately shows as invalid on the next scan attempt.

## Data Security

### What We Collect (Minimal)
| Data | Required? | Purpose |
|------|-----------|---------|
| Cardholder name | Optional | Displayed on verification screen for identity check |
| Scan timestamp | Automatic | Usage tracking and fraud detection |
| Business name (at scan) | Automatic | Per-sponsor analytics |
| Device push token | Automatic (Apple Wallet) | Sending card updates |
| IP address (at scan) | Automatic | Fraud detection only |

### What We Do NOT Collect
- No email addresses
- No phone numbers
- No payment information
- No location data (GPS)
- No browsing history
- No contacts
- No photos
- No social media accounts

### Encryption
- All communication between Apple Wallet and our server uses **HTTPS (TLS 1.3)**
- Authentication tokens are transmitted only over encrypted connections
- NFC communication uses **RSA encryption** with Apple's secure element
- The server runs on Railway.app's infrastructure with automatic SSL certificates

### Data Storage
- Card data and scan logs are stored in a **SQLite database** on the server
- The database is not publicly accessible — only the server can read/write
- No data is stored on third-party analytics platforms
- No data is shared with advertising networks

### Data Retention
- Scan data is kept for the card's validity period + 12 months for reporting
- When a card expires or is deactivated, device push tokens are deleted
- After the retention period, all data is permanently deleted
- Cardholders can request deletion of their data at any time

## Sponsor Data Access

### What Sponsors Can See
Each sponsor gets a private login showing **only their own business's data**:
- Total visits from cardholders
- Number of unique card IDs (not names)
- Visit dates and times
- Anonymized card IDs (e.g., DN-8F3A2B1C)

### What Sponsors Cannot See
- Other sponsors' data
- Cardholder names (the dashboard shows card IDs, not names)
- Cardholder contact information
- Scan data from other businesses
- Admin controls (deactivation, promotions, etc.)

### Sponsor Access Codes
- Each sponsor receives a unique **6-character alphanumeric code**
- Codes are case-insensitive
- No passwords to remember, no accounts to create
- Codes can be regenerated by the admin if compromised

## Admin Security

### Admin Dashboard
- The admin dashboard at `/admin` provides full access to all cards, scans, promotions, and POS configurations
- **Current state**: No authentication on the admin dashboard (planned for production)
- **Before production launch**: Basic authentication (username/password) will be added to protect admin routes

### Recommended Admin Security (for production)
- Add HTTP Basic Auth or session-based auth to `/admin` and `/api/*` routes
- Restrict admin access to a whitelist of IP addresses
- Use a strong, unique password
- Consider adding 2FA for admin access

## POS Integration Security

### API Credentials
- POS API keys (Toast, Square, Clover, etc.) are stored as **environment variables** on the server — never in code or git
- Each business's credentials are isolated — one business's keys cannot access another's POS
- Credentials are transmitted over HTTPS only
- We never store credit card or payment data

### Scan Verification
- When a POS scans the card, our server validates the card in real-time
- The server checks: is the card active? is it expired? is it flagged?
- The POS receives a JSON response with the validation result — no sensitive data is included
- Invalid or deactivated cards are immediately rejected

---

## Infrastructure Security

### Railway Hosting
- Hosted on **Railway.app** — a SOC 2 compliant cloud platform
- Automatic SSL/TLS certificates (HTTPS enforced)
- Server is isolated in its own container
- Automatic security updates via nixpacks builder
- No public SSH or database access

### Code Repository
- Source code is on **GitHub** in a private-capable repository
- Placeholder certificates are in the repo (for development only)
- Real Apple signing certificates are stored as Railway environment variables, never committed to code
- Dependencies are audited for known vulnerabilities

## Compliance

### CCPA (California Consumer Privacy Act)
- Cardholders have the right to know what data we collect (documented in Privacy Policy)
- Cardholders can request deletion of their data
- We do not sell personal information
- We do not share personal information with third parties for marketing

### Children's Privacy
- The card may be used by minors (high school students)
- We minimize data collection — only an optional name is collected
- No email, phone, or other personal information is collected from minors
- Parents/guardians can request deletion of their child's data

### Apple Wallet Guidelines
- Passes are signed with Apple-issued certificates
- We follow Apple's Wallet programming guidelines for pass structure and web services
- Push notifications use Apple's APNs (Apple Push Notification service) through official channels
